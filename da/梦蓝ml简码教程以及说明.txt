梦蓝式简码教程以及说明

一、用途
本文件用于说明 advancedmod 当前使用的 NPC 对话简码写法。
对话数据源默认写在:
c:\Users\nqwer\Desktop\da\src\examplemod\client\menglan\quanbu.txt

本简码的设计目标:
1. 让开发者直接写 txt 就能扩展 NPC 对话
2. 支持常驻选项、好感度选项、变量语组、连续回复、命令执行
3. 让后续扩展新变量时，尽量只补变量判断，不重写整套对话系统


二、文件头规则
文件头用于限制这一整段对话属于哪些 NPC。

写法:
allnpc :false
npcname:
npcuuid:
npcjob: all

说明:
1. allnpc:true
表示这一整段对话对所有 NPC 生效

2. allnpc:false
表示要继续看下面的名字、UUID、职业限制

3. npcname:
填 NPC 名字后，只对这个名字的 NPC 生效
留空表示不按名字限制

4. npcuuid:
填 NPC UUID 后，只对这个 UUID 的 NPC 生效
留空表示不按 UUID 限制

5. npcjob:
填职业名后，只对该职业 NPC 生效
填 all 表示全部职业都能用
留空表示只允许无职业或失业 NPC 使用


三、基础结构
一段完整对话通常由三部分组成:
1. NPC 开场语
2. 玩家可选语块
3. NPC 回复语块

基础示例:
npc-->有什么问题吗?;
me--> 1={今天天气怎么样?};
      2={你喜欢这里吗?};
npc {player(1)-->天气挺好的今天
      player(2)-->喜欢死了
    }

含义:
1. npc-->xxx;
表示 NPC 开场对白

2. me--> ...
表示下面开始写玩家选项

3. npc { ... }
表示下面开始写 NPC 回复块


四、常驻语块
常驻语块不会受到好感度门槛影响。
除了文件头目录限制外，它们始终存在。

写法:
1={要问的话};
2={要问的话};<+1>
3={要问的话};<-1>

标准结构:
常驻语块编号={要问的话};<加减好感度>

说明:
1. 常驻语块编号
例如 1、2、3、4

2. {要问的话}
表示玩家点击按钮时显示的文本

3. <加减好感度>
可选项
如果写了，就会在点击后增加或减少好感度
如果没写，默认就是 <0>

示例:
1={今天天气怎么样?+1好感};<+1>
2={你喜欢这里吗?-1好感};<-1>
3={文字弹出效果测试123};

解释:
1. 第一条是常驻语块，点击后加 1 好感
2. 第二条是常驻语块，点击后减 1 好感
3. 第三条没有写 <...>，默认就是 <0>，不加也不减


五、好感度语组(若要扩展需要先定义)
当 [] 里面直接写数字时，当前实现会把它识别为好感度门槛语组。

写法:
[1](1)={好感度1测试};
[5](2)={好感度5测试};

标准结构:
[需要的好感度](语块编号)={说的话};

说明:
1. [1]
表示当前好感度必须大于等于 1，这个选项才会显示

2. (1)
表示这个语块编号是 1

3. 这个语块不会去读默认的 player(1)
而是会去读 playerlove(1)

对应回复:
playerlove(1)--> 好感等级1
playerlove(2)--> 好感等级5

因此:
[1](1)={好感度1测试};
会对应:
playerlove(1)--> ...

[5](2)={好感度5测试};
会对应:
playerlove(2)--> ...


六、变量语组
为了以后扩展别的变量系统，内部结构已经预留了“变量语组”概念。

变量语组的设计思想:
[] 里放变量组名
() 里放语块编号
点击后只读取对应组名的 playerxxx(编号) 回复

概念示例:
[A](1)={口渴程度例子};

如果 A 代表口渴变量组，那么我们定义它的npc输出语句为playerkoke,那么输出就是:
playerkoke(1)--> ...

也可以写成:
[A]+(1)={口渴程度例子};

当前说明:
1. 当前项目已实装并真正接通的变量语组是好感组 love
2. 也就是 [数字](编号) 这类写法会自动按好感门槛处理，并转到 playerlove(编号)
3. 通用变量组的回复分组结构已经预留
4. 但如果要让例如口渴、饥饿、疲劳这类变量真正控制显示条件，还需要代码侧提供对应变量值和判断逻辑


七、NPC 回复块(也叫NPC输出块)
NPC 回复块默认前缀是 player。

默认回复写法:
player(1)-->天气挺好的今天
player(2)-->喜欢死了

好感语组回复写法:
playerlove(1)-->好感等级1
playerlove(2)-->好感等级5

通用变量语组回复写法:
player组名(编号)-->回复内容

例如:
playerkoke(1)-->你看起来很渴

含义:
如果某个选项属于 koke 这个变量语组，并且编号是 1，
那么点击后只会输出:
playerkoke(1)
不会输出:
player(1)
也不会输出:
playerlove(1)


八、连续回复块
一个玩家选项可以让 NPC 连续说多句。

写法:
player(4)-->{第一句
             第二句
             第三句
            }

示例:
player(4)-->{错误示范("give player apple 1")
             错误示范->("give player apple 1")<-
             正确示范 ->("give player apple 1")<-
             好了
            }

说明:
1. 花括号内每一行都会按顺序显示
2. 每一行都可以带命令后缀
3. 界面里会按“下一句”依次播放


九、命令后缀
NPC 回复中的某一句后面可以执行命令。
唯一生效格式:
文本 + 空格 + ->("命令")<-
正确示例:
正确示范 ->("give player apple 1")<-

错误示例:
错误示范("give player apple 1")
错误示范->("give player apple 1")<-
("give player apple 1")
(give player apple 1)

注意:
1. 文本和 ->("命令")<- 中间必须有空格
2. 没有这个空格就只会被当普通文本显示，不执行命令
3. 当前实现中，对话命令统一按单人世界高权限命令源执行
4. 如果当前不是单人世界，会提示无法执行对话命令


十、好感度规则
当前好感度规则如下:
1. 默认值是 0
2. 最低不会低于 0
3. 最高不会高于 100
4. 常驻语块后面的 <+数字> 或 <-数字> 会在点击时立即生效
5. [数字](编号) 这种写法使用的是“当前点击前的好感度”来决定该不该显示和该走哪组回复


十一、当前推荐写法
推荐按照下面结构写:

allnpc :false
npcname:
npcuuid:
npcjob: all

npc-->有什么问题吗?;
me--> 1={今天天气怎么样?+1好感};<+1>
      2={你喜欢这里吗?-1好感};<-1>
      3={文字弹出效果测试123};
      4={给予测试};
      [1](1)={好感度1测试};
      [5](2)={好感度5测试};
npc {player(1)-->天气挺好的今天
      player(2)-->喜欢死了
      player(3)-->啊啊啊啊啊啊啊啊啊啊啊啊啊啊啊啊abcd
      player(4)-->{错误示范("give player apple 1")
                   错误示范->("give player apple 1")<-
                   正确示范 ->("give player apple 1")<-
                   好了
                  }
      playerlove(1)--> 好感等级1
      playerlove(2)--> 好感等级5
    }


十二、常见理解方式
可以把它理解成三层:

第一层，玩家选项层
决定玩家能看到什么，点什么

第二层，条件层
决定某个选项是常驻语块，还是变量语组，是否需要门槛

第三层，NPC 输出层
决定点了以后去读 player、playerlove 或 playerxxx 哪一组回复


十三、给后续开发者的话
1. 如果只是做普通对话，直接写常驻语块 + player(编号) 就够了
2. 如果要做恋爱/好感分支，用 [数字](编号) + playerlove(编号)
3. 如果以后要做口渴、疲劳、饥饿等系统，请按“变量语组”思路扩展
4. 扩展新变量时，优先补变量判断与显示条件，不要破坏常驻语块和好感语组的兼容
5. 修改 txt 后，开发环境可通过 simcht r 重新加载
