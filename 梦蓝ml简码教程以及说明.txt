梦蓝式简码教程以及说明

一、用途
本文件用于说明 advancedmod 当前使用的 NPC 对话简码写法，以及 pool 新语法。
旧式梦蓝对话数据默认写在:
c:\Users\nqwer\Desktop\baritone-api-fabric-1.15.0\src\examplemod\assets\advancedmod\client\menglan\quanbu.txt
pool 全局语言池默认写在:
c:\Users\nqwer\Desktop\baritone-api-fabric-1.15.0\src\examplemod\assets\advancedmod\client\PUHS\pool.txt

本简码的设计目标:
1. 让开发者直接写 txt 就能扩展 NPC 对话
2. 支持常驻选项、好感度选项、变量语组、连续回复、命令执行
3. 让后续扩展新变量时，尽量只补变量判断，不重写整套对话系统
4. 让 pool 语法能独立做全局随机互动，不和旧式定制对话冲突


二、文件头规则
本节主要说明 menglan\quanbu.txt 这套旧式定制对话的文件头规则。
这一套文件现在只用于“定制指定 NPC”，不再用于覆盖所有 NPC。

写法:
allnpc :false
npcname:
npcuuid:
npcjob: all

说明:
1. allnpc:true
当前已无效化
写 true 或 false 都不会再让 quanbu.txt 直接对所有 NPC 生效

2. allnpc:false
当前同样无效化
保留这个字段只是为了兼容旧写法

3. npcname:
填 NPC 名字后，只对这个名字的 NPC 生效
留空表示不按名字限制

4. npcuuid:
填 NPC UUID 后，只对这个 UUID 的 NPC 生效
留空表示不按 UUID 限制

5. npcjob:
填职业名后，只对该职业 NPC 生效
填 all 现在也已无效化，不再代表全部职业都能用
留空表示只允许无职业或失业 NPC 使用

当前实际规则总结:
1. quanbu.txt 默认不会覆盖所有 NPC
2. 想定制某个 NPC，优先填写 npcname 或 npcuuid
3. npcjob 只有填具体职业名时才有“指定职业限制”的作用
4. 如果 npcname、npcuuid、npcjob 都不提供有效限制，这段定制对话默认不会匹配全部 NPC


三、基础结构
一段完整对话通常由三部分组成:
1. NPC 开场语
2. 玩家可选语块
3. NPC 回复语块

基础示例:
npc-->有什么问题吗?;
me--> 1={今天天气怎么样?};
      2={你喜欢这里吗?};
npc {player(1)-->天气挺好的今天
      player(2)-->喜欢死了
    }

含义:
1. npc-->xxx;
表示 NPC 开场对白

2. me--> ...
表示下面开始写玩家选项

3. npc { ... }
表示下面开始写 NPC 回复块


四、常驻语块
常驻语块不会受到好感度门槛影响。
除了文件头目录限制外，它们始终存在。

写法:
1={要问的话};
2={要问的话};<+1>
3={要问的话};<-1>

标准结构:
常驻语块编号={要问的话};<加减好感度>

说明:
1. 常驻语块编号
例如 1、2、3、4

2. {要问的话}
表示玩家点击按钮时显示的文本

3. <加减好感度>
可选项
如果写了，就会在点击后增加或减少好感度
如果没写，默认就是 <0>

示例:
1={今天天气怎么样?+1好感};<+1>
2={你喜欢这里吗?-1好感};<-1>
3={文字弹出效果测试123};

解释:
1. 第一条是常驻语块，点击后加 1 好感
2. 第二条是常驻语块，点击后减 1 好感
3. 第三条没有写 <...>，默认就是 <0>，不加也不减


五、好感度语组(若要扩展需要先定义)
当 [] 里面直接写数字时，当前实现会把它识别为好感度门槛语组。

写法:
[1](1)={好感度1测试};
[5](2)={好感度5测试};

标准结构:
[需要的好感度](语块编号)={说的话};

说明:
1. [1]
表示当前好感度必须大于等于 1，这个选项才会显示

2. (1)
表示这个语块编号是 1

3. 这个语块不会去读默认的 player(1)
而是会去读 playerlove(1)

对应回复:
playerlove(1)--> 好感等级1
playerlove(2)--> 好感等级5

因此:
[1](1)={好感度1测试};
会对应:
playerlove(1)--> ...

[5](2)={好感度5测试};
会对应:
playerlove(2)--> ...


六、变量语组
为了以后扩展别的变量系统，内部结构已经预留了“变量语组”概念。

变量语组的设计思想:
[] 里放变量组名
() 里放语块编号
点击后只读取对应组名的 playerxxx(编号) 回复

概念示例:
[A](1)={口渴程度例子};

如果 A 代表口渴变量组，那么我们定义它的npc输出语句为playerkoke,那么输出就是:
playerkoke(1)--> ...

也可以写成:
[A]+(1)={口渴程度例子};

当前说明:
1. 当前项目已实装并真正接通的变量语组是好感组 love
2. 也就是 [数字](编号) 这类写法会自动按好感门槛处理，并转到 playerlove(编号)
3. 通用变量组的回复分组结构已经预留
4. 但如果要让例如口渴、饥饿、疲劳这类变量真正控制显示条件，还需要代码侧提供对应变量值和判断逻辑


七、NPC 回复块(也叫NPC输出块)
NPC 回复块默认前缀是 player。

默认回复写法:
player(1)-->天气挺好的今天
player(2)-->喜欢死了

好感语组回复写法:
playerlove(1)-->好感等级1
playerlove(2)-->好感等级5

通用变量语组回复写法:
player组名(编号)-->回复内容

例如:
playerkoke(1)-->你看起来很渴

含义:
如果某个选项属于 koke 这个变量语组，并且编号是 1，
那么点击后只会输出:
playerkoke(1)
不会输出:
player(1)
也不会输出:
playerlove(1)


八、连续回复块
一个玩家选项可以让 NPC 连续说多句。

写法:
player(4)-->{第一句
             第二句
             第三句
            }

示例:
player(4)-->{错误示范("give player apple 1")
             错误示范->("give player apple 1")<-
             正确示范 ->("give player apple 1")<-
             好了
            }

说明:
1. 花括号内每一行都会按顺序显示
2. 每一行都可以带命令后缀
3. 界面里会按“下一句”依次播放


九、命令后缀
NPC 回复中的某一句后面可以执行命令。
唯一生效格式:
文本 + 空格 + ->("命令")<-
正确示例:
正确示范 ->("give player apple 1")<-

错误示例:
错误示范("give player apple 1")
错误示范->("give player apple 1")<-
("give player apple 1")
(give player apple 1)

注意:
1. 文本和 ->("命令")<- 中间必须有空格
2. 没有这个空格就只会被当普通文本显示，不执行命令
3. 当前实现中，对话命令统一按单人世界高权限命令源执行
4. 如果当前不是单人世界，会提示无法执行对话命令


十、好感度规则
当前好感度规则如下:
1. 默认值是 0
2. 最低不会低于 0
3. 最高不会高于 100
4. 常驻语块后面的 <+数字> 或 <-数字> 会在点击时立即生效
5. [数字](编号) 这种写法使用的是“当前点击前的好感度”来决定该不该显示和该走哪组回复


十一、当前推荐写法
如果你写的是 menglan\quanbu.txt，推荐按照下面结构写:

allnpc :false
npcname:
npcuuid:
npcjob:

npc-->有什么问题吗?;
me--> 1={今天天气怎么样?+1好感};<+1>
      2={你喜欢这里吗?-1好感};<-1>
      3={文字弹出效果测试123};
      4={给予测试};
      [1](1)={好感度1测试};
      [5](2)={好感度5测试};
npc {player(1)-->天气挺好的今天
      player(2)-->喜欢死了
      player(3)-->啊啊啊啊啊啊啊啊啊啊啊啊啊啊啊啊abcd
      player(4)-->{错误示范("give player apple 1")
                   错误示范->("give player apple 1")<-
                   正确示范 ->("give player apple 1")<-
                   好了
                  }
      playerlove(1)--> 好感等级1
      playerlove(2)--> 好感等级5
    }

如果你要做指定职业定制，可以写成:

allnpc :false
npcname:
npcuuid:
npcjob: 奶酪场工人

含义:
1. allnpc 只是占位，不参与判断
2. npcjob 写具体职业名时，才表示只给该职业 NPC 使用
3. 不推荐再写 npcjob: all，因为当前实现中它已不再表示“全部职业”


十二、常见理解方式
可以把它理解成三层:

第一层，玩家选项层
决定玩家能看到什么，点什么

第二层，条件层
决定某个选项是常驻语块，还是变量语组，是否需要门槛

第三层，NPC 输出层
决定点了以后去读 player、playerlove 或 playerxxx 哪一组回复


十三、给后续开发者的话
1. 如果只是做普通对话，直接写常驻语块 + player(编号) 就够了
2. 如果要做恋爱/好感分支，用 [数字](编号) + playerlove(编号)
3. 如果以后要做口渴、疲劳、饥饿等系统，请按“变量语组”思路扩展
4. 扩展新变量时，优先补变量判断与显示条件，不要破坏常驻语块和好感语组的兼容
5. 修改 txt 后，开发环境可通过 simcht r 重新加载


十四、pool 语言池新语法
pool 语法是独立于 quanbu.txt 的全局随机互动语法。
它的目标是:
1. 不用手写固定编号
2. 直接从语言池里随机抽选项
3. 让好感度变化和显示条件绑定到语池本身

数据文件:
c:\Users\nqwer\Desktop\baritone-api-fabric-1.15.0\src\examplemod\assets\advancedmod\client\PUHS\pool.txt

文件头:
allnpc :true
npcname:
npcuuid:
npcjob: all

说明:
1. pool.txt 是全局语池
2. 当前实现里它可以继续作为全局池使用
3. 如果填写名字、UUID、职业，也可以额外限制只给特定 NPC 使用

主语法:
pool+1:
pool-1:
pool0:

也兼容:
pool=0:

含义:
1. pool+1
表示该池中的句子被点击后，好感度 +1
2. pool-1
表示该池中的句子被点击后，好感度 -1
3. pool0 或 pool=0
表示该池中的句子被点击后，好感度不变

池内条目写法:
- 玩家说的话 | NPC 回复的话

示例:
pool+1:
    - 你声音真好听呀 | 谢谢夸奖，我很开心

这一条的含义是:
1. 玩家看到的选项是“你声音真好听呀”
2. 点击后 NPC 回复“谢谢夸奖，我很开心”
3. 同时好感度 +1

十五、pool 条件判断
pool 支持 if 语句来控制哪一批句子能进入随机池。

当前常用写法:
if >=<0>:
if >=<1>:
if >=<2>:

语义:
表示“当前好感度 >= 某个值”时，这一组条目才允许被随机抽到

例如:
pool+1:
    if >=<0>:
        - 你声音真好听呀 | 谢谢呀
    if >=<2>:
        - 你真可爱，好喜欢 | （偷偷的笑）笨蛋~

含义:
1. 当好感度 >= 0 时，第一条可抽
2. 当好感度 >= 2 时，第二条才会加入可抽范围

十六、pool 抽取逻辑
当前实现的抽取规则如下:
1. 每次对话会随机显示 3 个选项
2. 如果当前可用的 pool+1 不为空，系统会优先保证至少抽到 1 个 pool+1
3. 剩余位置会从当前全部可用池子里随机补满
4. 如果当前没有任何可用的 pool+1，就只从所有可用池子里随机抽取
5. 玩家选完一句并返回选项页后，会按更新后的好感度重新抽一轮

这里的“可用”指:
1. 满足文件头中的 NPC 限制
2. 满足 if 条件

十七、pool 与旧语法的关系
可以把当前系统理解成两套:
1. quanbu.txt
用于给指定 NPC 写“定制固定对话”
2. pool.txt
用于写“全局随机互动语池”

目前界面层会优先尝试读取 pool 语法。
如果当前没有可用的 pool 结果，再回退到 quanbu.txt 的旧式定制对话。

十八、pool 命令后缀
pool 右侧的 NPC 回复文本，同样支持旧系统的命令后缀:
文本 + 空格 + ->("命令")<-

例如:
- 给你一个苹果 | 收下吧 ->("give player apple 1")<-

含义:
1. NPC 会显示“收下吧”
2. 同时执行后面的命令
3. 文本和命令后缀之间必须保留空格
